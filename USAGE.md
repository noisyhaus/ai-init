# ai-init Usage

This guide is the concise command reference for the public `ai-init` install. In practice, you install a Codex skill set plus one local scaffold command, then enter through the skill that matches the current lane inside Codex.

For installation details, see [docs/install.md](docs/install.md).

## Install From Source

```sh
git clone https://github.com/noisyhaus/ai-init.git
cd ai-init
```

Install from the cloned repository:

```sh
./install.sh
```

This installs:

- `~/.codex/skills/ai-init`
- `~/.codex/skills/ai-init-start-work`
- `~/.codex/skills/ai-init-feature-addition`
- `~/.codex/skills/ai-init-bugfix`
- `~/.codex/skills/ai-init-session-recovery`
- `~/.codex/skills/ai-init-session-close`
- `~/.codex/skills/ai-init-finish-work`
- `~/.local/bin/ai-init`

## Basic Flow

From the project you want to initialize:

1. Start Codex in that directory.
2. Type the command that matches the lane.
3. Let the selected skill follow the shared ai-init workflow.

Common entrypoints:

- `$ai-init`
- `$ai-init-start-work`
- `$ai-init-feature-addition`
- `$ai-init-bugfix`
- `$ai-init-session-recovery`
- `$ai-init-session-close`
- `$ai-init-finish-work`

You can still run the CLI directly when you want the raw scaffold output:

```sh
ai-init
```

## Commands

```sh
ai-init
ai-init --new-project
ai-init --existing-project
ai-init --auto
ai-init --print-prompt
ai-init --with-rules
ai-init --with-memory-plus
ai-init --with-local-recall
ai-init --force
```

## Options

- `ai-init` - create core files and directories without overwriting existing files; auto-detect new vs existing project for the next-step prompt.
- `--new-project` - force new-project follow-up guidance.
- `--existing-project` - force existing-project follow-up guidance.
- `--auto` - auto-detect mode from current files and folders. This is the default.
- `--print-prompt` - print only the recommended follow-up prompt.
- `--with-rules` - also create starter rule files under `docs/ai/rules/`.
- `--with-memory-plus` - also create `docs/ai/agent_memory.md` and `docs/ai/user_preferences.md`.
- `--with-local-recall` - also create `.ai-index/` scaffolding and `scripts/ai-index.py` for local SQLite FTS recall. This implies `--with-memory-plus`.
- `--force` - overwrite starter files. Do not use this inside the `ai-init` repository root.

## Skill Workflow

After running `./install.sh`, Codex can see both the bootstrap skill and the direct lifecycle skills.

- `$ai-init` - bootstrap and orient a project.
- `$ai-init-session-recovery` - recover context at the start of a new chat or resumed task.
- `$ai-init-start-work` - prepare the Git lane before implementation.
- `$ai-init-feature-addition` - write feature spec and plan artifacts before coding.
- `$ai-init-bugfix` - reproduce, diagnose, fix, and verify a bug.
- `$ai-init-session-close` - close a session from Git evidence.
- `$ai-init-finish-work` - wrap a completed lane for commit, push, or merge readiness.

The direct lifecycle skills are thin wrappers over the shared references in `skills/ai-init/references/`, so the package keeps one workflow contract while exposing multiple public commands.

See [docs/skill-contracts.md](docs/skill-contracts.md) for the detailed lifecycle contracts.

## Session Recovery Output

Recovery should separate durable docs from live workspace evidence:

- Current objective
- Confirmed from docs
- Confirmed from live workspace
- Unknown or risky points
- Next smallest steps
- Verification plan

## Session Close Behavior

Session close should:

- collect Git evidence first with `git status --short`, `git diff --stat`, and `git diff --name-only`;
- separate in-scope changes from pre-existing dirt, generated noise, and external-path changes;
- read actual diff content only for in-scope lane files;
- update `docs/ai/current_state.md`;
- write `docs/ai/handoffs/YYYY-MM-DD.md` when useful;
- update durable memory files only when they exist and the lesson is stable;
- recommend staging or commit scope without silently committing or pushing.
